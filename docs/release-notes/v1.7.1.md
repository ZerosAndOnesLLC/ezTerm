# ezTerm v1.7.1

Patch on top of v1.7.0: fixes terminal paste mangling text copied from Windows apps (Claude Code GUI, Slack, browsers, anything that puts `\r\n` line endings on the clipboard).

## What changed

### Terminal paste — fix

- **`ui/components/terminal.tsx`** — `doPaste` used to UTF-8 encode the clipboard text and ship the raw bytes to the PTY. Two things this missed:
  - **Line-ending normalisation.** Windows clipboards carry `\r\n`. Terminals treat Enter as a single `\r` byte. Sending the raw `\r\n` submitted every line twice — once for the `\r`, once for the `\n` — which inserted a blank prompt between every pasted line and made indentation snowball under shells that auto-indent.
  - **Bracketed paste.** Modern shells (bash, zsh, fish), editors (vim, nano), and TUIs (less, claude-code's CLI) advertise DEC mode 2004 so the terminal can wrap pasted text in `\e[200~ … \e[201~`. The shell then knows the bytes are pasted and disables auto-indent / history expansion / inline command execution. Writing raw bytes skipped that, so the shell processed each character as if the user had typed it.
- Paste now flows through xterm.js's own `term.paste(text)`, which normalises `\r\n` / `\n` / lone `\r` all to a single `\r` and wraps the payload in bracketed-paste sequences whenever the remote side has the mode on. The pasted bytes then ride through xterm's `onData` listener that already pipes to the PTY — no second code path, no separate IPC.
