# ezTerm v1.6.9

Patch on top of v1.6.0: sidebar polish, fixes the SFTP drop-in upload path that Chromium quietly broke, bumps the Tauri / windows-rs stack, and removes the OS-drag-out implementation that never worked through WebView2. Also re-aligns the workspace, Tauri config, and UI bundle version numbers so the installed app and the auto-updater agree on what's running.

## What changed

### Sidebar polish

- **`ui/components/sessions-sidebar.tsx`** — Dropped the green pulsing rail/tile that highlighted "auto-detected" sessions; the visual noise outweighed the signal. The new look is the same flat row treatment used for everything else.
- **`migrations/20260518120000_clear_autodetect_color.sql`** — One-off cleanup migration that clears the auto-set `#34d399` colour from existing WSL / Local rows so they pick up the default styling instead of carrying the now-removed accent.

### SFTP drop-in upload — fix (regression from a Chromium change)

- **`ui/components/sftp-pane.tsx`** — Two changes against the same Chromium quirk:
  - The drop handler now snapshots every `DataTransferItem` entry **synchronously**, before any `await`. Recent Chromium versions invalidate the `items` collection as soon as the drop handler yields, so an `await` mid-handler used to leave the upload code holding a stale list and silently dropping files.
  - `dragover` now sets `dropEffect = 'copy'`. WebView2 won't fire `drop` for OS-sourced files unless a copy effect is advertised — the old version got drops from in-app sources but ignored OS file-manager drops.

### Drag-out — removed

- **`src-tauri/src/sftp/drag/`** (deleted) — The OLE `IDataObject` / `IDropSource` / `DoDragDrop` implementation for streaming SFTP drag-out is gone, along with the `sftp_drag` Tauri command and the `dragTestFile` test stub.
  - `DoDragDrop` returned `E_FAIL` from every path we tried: worker thread, Tauri main thread via `run_on_main_thread`, a fresh top-level Win32 window in the same process, a separate sidecar process with foreground + active + mouse held, and inside a real `WndProc` triggered by `SendMessage WM_LBUTTONDOWN`.
  - HTML5 `dragstart` + `DownloadURL` produces no OS drag in WebView2 either.
  - Conclusion: Tauri / WebView2 on Windows has no working drag-source path today. **Right-click → Download…** (already wired) is the supported UX.
  - `src-tauri/src/splash.rs` doc comment updated to reflect the limitation.

### Dependencies

- `tauri` 2.11.1 → **2.11.2** (and the matching `@tauri-apps/api` line)
- `windows-rs` / `windows-core` 0.58 → **0.62** — `BOOL` moved into `windows::core`, and `IDataObject::DAdvise` / `IStream::CopyTo` now take `Ref<T>` instead of `Option<&T>`. (These no longer compile against any of the `sftp/drag/` code, but that's deleted in this release.)
- `@types/node` patch bump.
- ESLint pinned to **9.x** — `eslint-config-next` still pulls in an `eslint-plugin-react` that's incompatible with ESLint 10.

### Version alignment

- **`Cargo.toml`** (workspace), **`src-tauri/tauri.conf.json`**, **`ui/package.json`**, and **`ui/package-lock.json`** all now report `1.6.9`. Earlier releases bumped only the workspace `Cargo.toml`, leaving `tauri.conf.json` and `ui/package.json` stuck at `1.3.7`. That caused the installed app's auto-updater to compare a stale version against `latest.json` and re-trigger passive reinstalls on every launch.
