# ezTerm v1.4.0

Minor: vault master-password management. The vault now supports changing the master password from inside the app, generating a one-time recovery code, unlocking with that recovery code, and a "forgot password" reset that wipes the encrypted blobs and starts fresh.

After the initial implementation we ran four parallel review agents (performance, security, completeness, code quality) and addressed every Critical, High, and Medium finding before this release. The notable hardenings:

- **`vault_reset` requires a server-validated confirmation token** (`DELETE-MY-VAULT`). The UI's "type DELETE" gate is no longer the only barrier; any rogue invoke from inside the webview is rejected at the Rust boundary.
- **Re-key is now race-free.** `vault_change_password` holds `state.vault.write()` for the entire operation and pulls every credential / sync-blob read inside a `BEGIN IMMEDIATE` transaction. A concurrent `credential_create`, sync writer, or SSH connect can no longer observe a torn rekey or wedge a row encrypted under the old key into the rotated DB.
- **Recovery codes are single-use.** A successful `vault_unlock_with_recovery` clears the wrap on disk, so a leaked code is no longer a permanent backdoor. Generating a new code requires master-password re-verify and explicit confirmation when overwriting an existing wrap. The dialog also takes a snapshot before regenerating, and never auto-generates on mount.
- **`vault_unlock_with_recovery` is constant-time-ish.** "No recovery code provisioned", "invalid character", "wrong code", and "AEAD mismatch" all run a full Argon2id derive and all surface as `BadPassword`. No timing or error-class oracle for the recovery state.
- **Cooldown applies to every Argon2 path.** `vault_unlock`, `vault_verify_password`, `vault_change_password` (old-password verify), and `vault_unlock_with_recovery` share the same failure counter and exponential lockout. Bypassing the cooldown by routing through a different command no longer works.
- **Vault rotation tears down active connections + sync target.** After a successful `change_password` or `reset`, every SSH/SFTP/local connection is closed and the in-memory sync target is cleared. Old credentials don't leak into post-rotation operations, and the sync writer doesn't keep PUTting to the previous remote with the old passphrase.
- **KDF parameter floor on unlock.** Stored `m_cost_kib` below 32 MiB or `t_cost` below 2 are treated as DB tampering and refused. Prevents an attacker with DB-write access from downgrading parameters and offline-cracking the verifier.
- **Snapshots are tightened.** Retention is now both count-capped (5) and time-capped (30 days). On unix the snapshot file is `chmod 0600` immediately after `VACUUM INTO`. The "snapshot before destructive op" is also taken on recovery-code regeneration, not just on password change.
- **Recovery code material is `Zeroizing`.** The decoded bytes, the canonical re-encoded form, and the generated code itself live in zeroizing wrappers instead of bare `Vec<u8>` / `String`.
- **Backend strength gate.** New master passwords must be ≥ 12 characters AND mix at least two character classes. The UI strength meter still nudges; the backend now also rejects trivial choices.
- **`reset` nulls both credential columns explicitly.** No longer relies on the `ON DELETE SET NULL` FK cascade.

## What changed

- **`migrations/20260515154320_vault_recovery.sql`** — Adds three optional columns to `vault_meta` (`recovery_salt`, `recovery_kdf_params`, `recovery_wrapped_key`) used to store the recovery wrap. NULL means "no recovery code provisioned."
- **`src-tauri/src/vault/mod.rs`** — Two new operations:
  - `change_password` re-derives a fresh key from the new password under the current default `KdfParams` (so a password change is also a security-parameter refresh) and re-encrypts every `credentials` row and every `sync.*_blob` setting in a single sqlx transaction. The verifier and salt are rewritten under the new key, the recovery wrap is invalidated, and the caller is expected to lock the vault.
  - `reset` wipes `vault_meta`, the `credentials` table, and the `sync.*_blob` settings; sessions stay but lose their `credential_id` reference. Single transaction so a partial wipe is impossible.
- **`src-tauri/src/vault/recovery.rs`** (new) — Recovery-code generation and unlock. The code is a random 24-character base32 string (120 bits of entropy, hyphen-grouped for readability). At generation we Argon2id-derive a recovery key from the code and AEAD-wrap the current password-derived key under it; at unlock we re-derive, unwrap, and verify by decrypting the existing verifier. Transcription tolerances (lowercase, spaces, hyphens; common `0↔O`, `1↔I`, `8↔B` substitutions) are baked into the decoder.
- **`src-tauri/src/vault/snapshot.rs`** (new) — Pre-change DB snapshots via SQLite `VACUUM INTO`. Snapshots land at `<data_local>/backups/vault-<label>-<timestamp>.sqlite`; the five most recent are kept, older ones are pruned best-effort.
- **`src-tauri/src/commands/vault.rs`** — Five new Tauri commands: `vault_change_password`, `vault_recovery_status`, `vault_generate_recovery_code`, `vault_unlock_with_recovery`, `vault_reset`. The recovery-unlock path shares the unlock-failure cooldown with the password path.

## UI

- **`ui/components/change-password-dialog.tsx`** — Old / new / confirm fields with a live strength meter. On success shows the snapshot path and forces the user back through the unlock screen with the new password.
- **`ui/components/recovery-code-dialog.tsx`** — Generates a fresh code on open (regenerating invalidates any prior code), shows it exactly once with copy-to-clipboard.
- **`ui/components/recovery-unlock-dialog.tsx`** + **`ui/components/reset-vault-dialog.tsx`** — Reachable from the unlock screen via "Use recovery code" (only shown when one is provisioned) and "Forgot password?" (always shown). The reset path requires typing `DELETE` as a confirmation.
- **`ui/components/unlock-screen.tsx`** — Wires the two new escape hatches into the lock screen, with a probe of `vault_recovery_status` so the recovery link only appears when it'd actually work.
- **`ui/components/sessions-sidebar.tsx`** — Adds "Change master password…" and "Recovery code…" entries to the root context menu (alongside Backup / Cloud sync).
- **`ui/lib/password-strength.ts`** (new) — Hoists the strength heuristic out of `unlock-screen.tsx` so the change-password dialog can reuse it.

## Notes

- Recovery codes are *invalidated by password changes*. The wrap is made over the password-derived key, so changing that key would orphan the wrap. The dialog warns the user to regenerate after a password change.
- The reset path does not require any password — premise is the user has lost theirs. Anyone with local file access already had read access to the encrypted blobs (they just couldn't decrypt them).
