# ezTerm v1.6.0

Minor: completes the in-process SFTP drag-and-drop story for Windows. v1.5.0 shipped per-folder drop targets, directory uploads, in-pane progress, and the OLE drag-source scaffold. This release lifts the 256 MB upload cap, ships real streaming drag-out from the SFTP listing, and adds multi-select on the SFTP pane so a user can drag a group of files at once. macOS and Linux drag-out remain TODO (issue #28 phases B5/B6).

## What's new

### A3 — streaming uploads (no cap)

- **`src-tauri/src/sftp/upload_stream.rs`** (new) — Each upload registers a writer task that holds the SFTP session lock for its lifetime and consumes chunks from a bounded mpsc channel. Per-chunk acks let the frontend stop slicing on the first remote write error (disk full, permission denied) rather than after the last byte ships.
- **Tauri commands**: `sftp_upload_begin` / `sftp_upload_chunk` / `sftp_upload_finish` / `sftp_upload_abort`. Backpressure: the writer channel is bounded so `chunk` awaits if the SFTP link is slower than the frontend.
- **`ui/components/sftp-pane.tsx`** — `uploadOneFile` now slices the dropped `File` into 1 MB chunks via `File.slice().arrayBuffer()` and pushes them through `sftpUploadBegin`/`Chunk`/`Finish`. The 256 MB cap is replaced with a 16 GB sanity guardrail.
- New `StreamingUploadStatus` strip in the pane shows live per-file progress driven by chunk acks; finished entries self-clear after a moment, errors stick for 4 seconds so the user can see what failed.

### B2 + B3 — streaming SFTP drag-out (Windows)

- **`src-tauri/src/sftp/drag/win.rs`** — New `SftpStream` (IStream) backed by a `tokio::sync::mpsc::Receiver`. A reader task on the tokio runtime opens the remote file, pushes 64 KiB chunks into the channel; `IStream::Read` (called by Explorer's file-copy worker) drains the channel via `Receiver::blocking_recv`. The OS file-copy dialog sees a real `STATSTG.cbSize` (from a pre-drag `stat` round-trip) so its progress bar and ETA work. Drag cancel propagates: dropping the IStream closes the channel, the reader task gets a send error and exits.
- **Tauri command**: `sftp_drag(connection_id, remote_paths)` — runs the blocking `DoDragDrop` on `spawn_blocking` to avoid starving the tokio runtime.
- **`ui/components/sftp-file-row.tsx`** — Mousedown-to-drag hysteresis (5 px) on non-folder rows. Crossing the threshold fires the OS-native drag; releasing without moving just selects.

### B4 — multi-file drag-out + selection UI

- **`SftpFileData`** in `win.rs` now stores `Vec<SftpFile>` and builds a `FILEGROUPDESCRIPTORW` sized for N entries; `GetData(CFSTR_FILECONTENTS, lindex=N)` routes to the Nth file's IStream. Each file gets its own reader task and channel; they queue on the SFTP session mutex and run in lindex order (which matches the order Explorer requests them).
- **`ui/components/sftp-pane.tsx`** — Standard file-manager selection idioms: plain click replaces, Ctrl/Cmd-click toggles, Shift-click extends a range from the last anchor. Selection clears on `cwd` change. Dragging from a selected row carries the entire selection (folders filtered out — directory drag-out is a separate phase since it needs recursive remote listing).
- Selected rows render with an accent tint so the user can see what's about to come along on a drag.

### Cross-platform stance

Drag-out is **Windows-only** in this release. On macOS / Linux the `sftp_drag` command returns a clear `not yet implemented on this platform` error and the UI suppresses that as a no-op (no scary toast). Issue #28 phases B5 (`NSFilePromiseProvider`) and B6 (XDS / `wl_data_source`) track the remaining native work; the React side won't need changes when they land.

### Deferred / not yet implemented

- **B5** — macOS drag-out (`NSFilePromiseProvider`).
- **B6** — Linux drag-out (X11 XDS, Wayland `wl_data_source`).
- **B7** — polish: custom drag cursor with progress, bandwidth limit, mid-drag SFTP disconnect recovery.
- **Recursive directory drag-out** — needs a remote walk to materialise a `FILEGROUPDESCRIPTORW` with one entry per descendant. Tracked separately from the file-selection multi-file path (#28 has a note).
