# ezTerm v1.9.0

Port forwards created from a tab's **Forwards** pane now show up immediately and can be **saved to the session** so they come back on the next connect. Previously a forward added from the pane was tab-only — it could stay invisible in the list until a reconnect, and there was no way to promote it into the session's saved config from the pane.

## What changed

### Per-tab forwards — instant display + persist (#118)

- **Shows immediately** — a forward started or created from the per-tab pane is folded into the list right away (and flips to *running*) instead of only appearing after the next status event or a reconnect. The runtime layer deliberately doesn't emit an initial "Running" event; the pane now upserts the awaited summary by `runtime_id` itself.
- **Save to this session** — the pane's *Add forward* dialog gains a **“Save to this session (persists across reconnects)”** option. When ticked, the forward is written to the session's stored forward config **and** started on the current tab in one step, so it's live now and auto-restored on the next connect. Ticking it also reveals the **auto-start on connect** toggle.
- **Dynamic (SOCKS) forwards round-trip correctly** end to end — the internally-tagged start target and the nested forward spec (both carrying a `kind` field) deserialize cleanly for ephemeral and persistent dynamic forwards alike.

> Editing and deleting saved forwards continues to live in the session editor's **Forwards** tab; deleting a tab forward is in the pane.

### Tests

- DB-layer coverage that local, remote, and dynamic forwards persist and list back, against an in-memory SQLite pool seeded with a real session row.
- Serde tests pinning the exact JSON the forward dialog sends — ephemeral dynamic (SOCKS), persistent-by-id, and a persistent dynamic `ForwardInput` — so a frontend/back-end shape drift fails the build instead of silently breaking a forward at runtime.
